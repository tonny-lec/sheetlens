# SheetLens 設計書

日付: 2026-07-07 / 状態: 承認済み（ユーザーレビュー待ち）

## 目的

業務 Excel（.xlsx / .xlsm）の構造と業務上の意味を、AI エージェントが誤読しない
中間表現に変換する CLI ツール。Excel 業務ツールの Web 化案件で、AI が要件定義・
設計を行う際の読み取り精度と業務理解を人間と同等に引き上げる。

## 解決する問題

生の xlsx を LLM に読ませると次の 2 層で失敗する:

1. **構造層の読み取り失敗** — 表構造の再現、数式、条件付き書式、セル結合、
   レイアウト、入力規則の認識漏れ
2. **意味層の欠落** — 手入力される表と VBA が自動入力する表の区別、プルダウン
   選択値の業務上の意味、セル・操作と業務フロー・業務状態の対応関係は、
   ファイル単体からは導出できない

## 確定した要件

| 項目 | 決定 |
|------|------|
| 実行環境 | 業務 PC（Windows 想定、開発ツールのインストールは自由） |
| 開発環境 | 開発 PC（Linux）。実サンプルは持ち込めないためダミーデータで開発 |
| 提供形態 | CLI。ファイルベースの中間表現を生成（特定の AI エージェントに非依存） |
| 意味層の収集 | AI エージェントが人間にインタビューし、回答を注釈ファイルに蓄積 |
| VBA | スコープに含む。ソースコード抽出まで（解釈は下流の AI に任せる） |
| テストデータ | ユーザーの記憶ベースで再現したダミー業務 Excel + 機能網羅フィクスチャ |
| 成功基準 | 中間表現のみを読んだ AI の QA 正答率（生 xlsx 読み取りとの比較で改善を定量化） |

## 技術選定

**Python 3.12+ / uv 管理。** 採用理由: openpyxl（構造抽出）と oletools（VBA 抽出）の
組み合わせが解析カバレッジ最大で、フィクスチャ生成も同一言語で完結する。

- 依存: openpyxl, oletools, pydantic, pyyaml, typer
- dev 依存: pytest
- openpyxl が拾えない要素が出た場合は、OOXML（ZIP + XML）を直接読む補完パーサを
  該当要素に限って後付けする（全面自作はしない）

不採用案: TypeScript CLI（SheetJS/ExcelJS は条件付き書式・VBA の読み取りが弱く、
本ツールの目的と弱点が正面衝突）、OOXML 全面自作パーサ（v1 には過剰）。

## アーキテクチャ

### 設計原則

1. **構造層 = 再生成可能、意味層 = 蓄積資産。** `structure/` は extract の再実行で
   完全に再生成される。`annotations/` は人間の知識であり、ツールは絶対に消さない。
2. **SheetLens 自体は LLM を呼ばない決定的ツール。** LLM を使う作業（インタビュー、
   解釈）は上に乗る AI エージェントの仕事。これにより出力が安定し、どの AI
   エージェントからも使える。
3. **静かな欠落の禁止。** 抽出できなかった情報を黙って落とすことを最大の障害と
   定義する（後述のエラー処理）。
4. **見た目の再現より、参照関係と意味の明示。** 出力は人間向けではなく LLM が
   コンテキストで消費する中間表現。評価基準は「LLM が誤読しないか」。

### 出力: 1 Excel = 1 プロジェクトディレクトリ

```
sheetlens extract 見積管理.xlsm  →  見積管理.sheetlens/

見積管理.sheetlens/
├── manifest.json        # 元ファイルのハッシュ・シート一覧・シート間依存グラフ・
│                        # extraction_gaps（抽出できなかった要素の記録）
├── structure/           # 【構造層】決定的に再生成される
│   ├── sheet-<シート名>.md
│   ├── vba/<モジュール名>.bas
│   └── raw.json         # 省略なしの全抽出データ（機械可読の正）
├── annotations/         # 【意味層】インタビュー回答の蓄積。再生成で消さない
│   └── <シート名>.yaml
├── questions.md         # 構造から自動生成した「人間に聞くべき質問リスト」
└── README.md            # AI エージェント向け入口。全体像・読み方・欠落警告
```

### データフロー（3 段階）

1. **extract（決定的）**: xlsx/xlsm → `structure/` + `questions.md` を生成
2. **インタビュー（AI + 人間）**: AI エージェントが questions.md を元に業務担当者へ
   質問し、回答を `annotations/*.yaml` に記録する
3. **compile**: 構造層 + 注釈を統合し、README.md と各シート Markdown に意味層を
   織り込んだ最終形を再生成する

Excel 更新時は extract を再実行。annotations/ は保持され、現構造に存在しなくなった
範囲への注釈は「孤立注釈」として警告リストに出す（自動削除しない）。

**生成物の位置づけの明確化**: `sheet-*.md` と `README.md` は常に生成物であり、
手編集しない。extract 直後は注釈なしのビュー、compile 後は注釈織り込み済みの
ビューになる。機械可読の正は `raw.json`（構造）と `annotations/`（意味）の 2 つで、
Markdown はこの 2 つから何度でも再生成できる。

## 中間表現フォーマット（構造層）

### シート Markdown（`sheet-<名前>.md`）の構成

1. **概要** — 使用範囲、結合セル数、数式セル数、入力規則数
2. **レイアウトマップ** — セル座標つきグリッド表現。結合セルは `[A1:H1 結合]` の
   ように範囲で明示
3. **領域** — ツールが検出したブロック（ヘッダ情報、明細テーブル等）。テーブルは
   列単位で数式パターンを記述
4. **数式（正規化済み）** — 同一パターンは「範囲 + パターン + 例外」に集約。
   例:「E11:E30 = C*D（行内参照）」。パターンから外れるセルは**例外として強調**
   （手修正された数式は業務上の特例やバグの発見に直結するため）
5. **入力規則** — プルダウンはリスト参照元と実際の選択肢値を展開して記載
6. **条件付き書式** — ルールを自然言語 + 数式の両方で記載
7. **VBA との接続** — ボタン・イベントとプロシージャの対応

### 二層構え

- Markdown は LLM 向けの**ビュー**。詳細が必要なら AI が `raw.json` を検索する
- シート間参照は manifest.json に依存グラフとして集約

## 意味層

### questions.md の生成ルール（v1 で実装する検出ルール）

| 構造上の検出 | 生成する質問 |
|--------------|--------------|
| 数式なしのデータ領域 | 誰が・いつ・何を見て入力するか（手入力 / 貼り付け / VBA 自動入力の区別） |
| 入力規則のリスト | 各選択肢の業務上の意味と、選択による後続処理・判断の変化 |
| VBA プロシージャとボタン | 押す業務タイミングと前提条件 |
| 条件付き書式 | 強調表示されたとき業務担当者は何をするか |
| シート単位 | シートの役割と業務フロー上の工程 |
| 保護・非表示（シート/列） | なぜ隠されて/守られているか |

### annotations YAML スキーマ

```yaml
sheet: 見積入力
role: "営業担当が見積作成時に使うメイン入力画面"
workflow_stage: "受注前・見積提示フェーズ"
targets:
  - range: A10:H30
    kind: input_source        # 列挙型（下記）
    value: manual             # manual / vba / paste_from_system / formula
    by: "営業担当"
    when: "顧客ヒアリング後"
  - range: C5
    kind: dropdown_semantics
    values:
      通常: "標準納期・標準価格を適用"
      特急: "納期短縮。Module1 が割増率を自動設定"
questions_answered: [q-001, q-004]
```

`kind` は固定語彙の列挙型: `input_source` / `dropdown_semantics` / `trigger_timing` /
`alert_action` / `sheet_role` / `free_note` / `hidden_reason`。自由記述は `free_note` で許容しつつ、
頻出パターンは構造化して AI が確実に拾えるようにする。

### compile での統合

- 各領域・要素の直下に `> 💬 業務上の意味:` ブロックとして注釈を織り込む
- 未回答質問は `> ❓ 未確認:` として残す。**「分かっていないことの明示」**により、
  AI が推測で要件を捏造することを防ぐ（生 xlsx 読み取りとの決定的な違い）

## CLI とコンポーネント

### コマンド（v1 は 3 つ）

```
sheetlens extract <file.xlsx|xlsm> [-o <dir>]   # 構造層 + questions.md を生成
sheetlens compile <dir>                          # 注釈を統合した最終 Markdown を再生成
sheetlens check <dir>                            # 孤立注釈・未回答質問・スキーマ違反を報告
```

### パッケージ構成

| モジュール | 責務 |
|-----------|------|
| `reader/` | openpyxl ラッパー + oletools の VBA 抽出。Excel を読む処理をここに閉じ込める |
| `model/` | 中間表現のデータモデル（pydantic）。reader と renderer の間の契約 |
| `detectors/` | 領域検出・数式パターン集約・質問生成。ルールごとに独立モジュール（今後最も増える部分） |
| `renderers/` | Markdown / raw.json / manifest.json の生成 |
| `annotations/` | YAML スキーマ検証・孤立注釈チェック |
| `cli.py` | typer エントリポイント |

## エラー処理

- **静かな欠落の禁止**: openpyxl が解釈できない要素・未対応の条件付き書式などは、
  処理を継続しつつ manifest.json の `extraction_gaps` に記録し、README.md 冒頭に
  「⚠ この抽出には N 件の欠落があります」と明示する
- 破損ファイル・パスワード保護ファイルは明確なエラーメッセージで停止する
- annotations YAML のスキーマ違反は、ファイル名・該当箇所つきでエラー報告する

## テストと評価（3 層）

1. **ユニット + ゴールデンテスト**: openpyxl で生成した機能網羅フィクスチャ
   （結合・数式・条件付き書式・入力規則・VBA…）→ 期待 Markdown のスナップショット
   比較。pytest で常時実行
2. **QA 評価ハーネス**: `eval/` にダミー業務 Excel（ユーザーの記憶ベースで再現）+
   正答つき質問セットを用意。AI エージェントに ①生 xlsx ②中間表現 のそれぞれ
   だけを渡して回答させ、正答率を比較する。**これが v1 の成功判定**
3. **実地検証**: 業務 PC で実ファイルに実行し、extraction_gaps と誤抽出を報告して
   もらい、フィクスチャに欠けていたパターンを逆輸入する改善ループ

## スコープ外（v1 では作らない）

- MCP サーバ / Claude Code スキルとしてのラッパー（CLI コアの上に将来足せる設計は維持）
- VBA コードの解釈・要約（抽出のみ。解釈は下流 AI の仕事）
- .xls（旧形式）対応、パスワード保護ファイルの解除
- グラフ・図形・ピボットテーブルの詳細抽出（存在の記録は manifest に含める）
- 複数 Excel ファイル横断の依存解析（ファイル外参照は「外部参照あり」として記録のみ）
